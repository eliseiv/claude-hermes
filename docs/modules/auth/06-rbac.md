# Auth — RBAC

## Принципалы
- Auth-эндпоинты — **до** получения пользовательской идентичности, поэтому RBAC в обычном смысле не применяется: `register`/`token`/`refresh`/`apple`/`jwks` не требуют пользовательского JWT и не проверяют `sub`.
- Авторизация заменена **rate-limit per IP** ([05-security.md](05-security.md)) — единственный контроль доступа к этим эндпоинтам.

## Изоляция
- Auth-контур **не пересекается** с admin-auth ([ADR-009](../../adr/ADR-009-admin-token-auth.md)): admin-токен здесь не принимается и не выдаётся; роли `admin` в выпускаемом JWT **нет**.
- `register`/`token`/`apple` **не принимают** `userId` в теле — идентичность назначает backend (uuid4 для нового устройства, find-by-device для известного, find-by-`apple_sub` для Apple-входа). Спуфинг чужого `userId` на этом контуре невозможен by construction; для Apple — `userId` резолвится из верифицированного `apple_sub`, не из клиентского ввода ([ADR-043](../../adr/ADR-043-sign-in-with-apple.md)).
- После выпуска токена вся последующая авторизация `/v1/*` — стандартная (`sub`-изоляция, сверка `userId==sub`), как и прежде.

## Выпускаемые роли
- Только `user` (владелец своих ресурсов). Эскалация до `admin` через выпущенный токен невозможна — admin-контур изолирован.
