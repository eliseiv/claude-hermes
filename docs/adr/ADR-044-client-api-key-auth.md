# ADR-044 — Авторизация клиентским API-KEY (`X-API-Key`) + идентичность через `X-User-Id`

- Статус: Accepted
- Дата: 2026-06-23
- Связан с: [ADR-007](ADR-007-lazy-user-provisioning.md) (lazy provisioning — сохраняется), [ADR-009](ADR-009-admin-token-auth.md) (admin-token — неизменен), [ADR-018](ADR-018-embedded-auth-issuer.md) / [ADR-043](ADR-043-sign-in-with-apple.md) (JWT/Apple — переводятся в «спящий» режим), [05-security.md](../05-security.md), [08-api-documentation.md](../08-api-documentation.md), [modules/api-gateway/](../modules/api-gateway/README.md)
- Расширяет/ревизует: модель аутентификации [05-security.md §Аутентификация](../05-security.md) (для клиентского контура)

## Context

Сервис разворачивается как backend для iOS-приложения, где Hermes подключается как полноценный агент (см. [ADR-045](ADR-045-hermes-as-agent-proxy.md)). Решением пользователя зафиксирована **единая клиентская модель авторизации**: один доверенный клиентский API-KEY (`X-API-Key`) на каждом эндпоинте, а идентичность пользователя передаётся отдельным заголовком `X-User-Id` (UUID).

Прежняя модель ([ADR-018](ADR-018-embedded-auth-issuer.md)): идентичность из RS256 JWT (`sub`), встроенный issuer (`/v1/auth/*`), device-based + Sign in with Apple ([ADR-043](ADR-043-sign-in-with-apple.md)). Эта модель остаётся в коде, но в новой поставке клиент её не использует.

Требуется зафиксировать:
- как меняется auth-контур (откуда берётся identity, какой секрет, какая зависимость);
- судьбу JWT/Apple-кода (не удаляем — «дремлет», чтобы не ломать БД-схему и тесты);
- security-схемы Swagger (`APIKeyHeader` вместо `bearerAuth`; `X-Admin-Token` уже `APIKeyHeader`);
- модель доверия к `X-User-Id`;
- отношение к действующим ADR-007/009.

Принципиальное решение (НЕ пересматривать): клиентский ключ — **доверенный** (хранится в доверенном клиентском окружении / control plane клиента), поэтому переданная им идентичность (`X-User-Id`) принимается на доверии.

## Decision

### 1. Клиентский контур — `X-API-Key` + `X-User-Id`

- Все пользовательские эндпоинты `/v1/*` (кроме health/metrics, preview, admin, adapty-webhook) авторизуются **единым клиентским ключом** `CLIENT_API_KEY`, передаваемым в заголовке `X-API-Key`. Сравнение — **constant-time** (`hmac.compare_digest`), как `_admin_token_matches` ([ADR-009](ADR-009-admin-token-auth.md)). Отсутствие/несовпадение → `401` (причина не раскрывается).
- Идентичность пользователя берётся из заголовка `X-User-Id` (UUID). Это субъект всех доменных операций: `users.id ≡ X-User-Id`. Отсутствие/невалидный UUID → `401` (нет субъекта).
- Новая функция `verify_client_api_key()` (`src/app/api_gateway/auth.py`) — constant-time проверка `X-API-Key` vs `settings.client_api_key`. Зависимость `get_current_user` (`src/app/deps.py`) переписывается: вместо `verify_bearer_token` (JWT) — `verify_client_api_key()` + извлечение `userId` из `X-User-Id`.
- **Ротация ключа** — поддержка двух активных значений на grace-период по образцу admin ([ADR-009 §5](ADR-009-admin-token-auth.md)): `CLIENT_API_KEY` (основной) + опц. `CLIENT_API_KEY_PREV`. Оба сравниваются constant-time, любое совпадение валидно.

### 2. Lazy provisioning сохраняется ([ADR-007](ADR-007-lazy-user-provisioning.md) — без изменений)

- `provision_user(session, user_id)` вызывается в `get_current_user` **после** успешной проверки `X-API-Key` и извлечения `userId` из `X-User-Id`, **до** любой FK-зависимой операции. Семантика идемпотентного upsert (`INSERT ... ON CONFLICT (id) DO NOTHING`) не меняется.
- Меняется **только источник** `userId`: раньше `sub` верифицированного JWT, теперь `X-User-Id` доверенного клиента. Инвариант «строка `users` гарантированно существует до FK-вставки» сохраняется. [ADR-007](ADR-007-lazy-user-provisioning.md) остаётся Accepted; этот ADR лишь подменяет канал доставки identity.

### 3. Модель доверия к `X-User-Id`

- **Идентичность доверяется, т.к. клиентский ключ доверенный.** Знание `CLIENT_API_KEY` = право действовать от имени любого `X-User-Id`. Это осознанный сдвиг модели угроз относительно [ADR-018](ADR-018-embedded-auth-issuer.md) (где `sub` был криптографически связан с подписью токена конкретного пользователя).
- Следствие: единый клиентский ключ — **высокопривилегированный секрет** (компрометация = доступ ко всем пользователям). Митигация: высокоэнтропийное значение (≥ 32 байта), secret manager, redaction (`X-API-Key` в денилист), TLS на edge, ротация (`CLIENT_API_KEY_PREV`), per-instance значение (мульти-инстанс, [ADR-017](ADR-017-shared-server-traefik-deploy.md)). Усиление (per-client ключи, mTLS, подпись `X-User-Id`) — [Q-044-1](../99-open-questions.md).
- **`require_owner` теряет смысл.** При едином ключе субъект = `X-User-Id` по определению; нет «тела с чужим `userId` против `sub`» (прежняя сверка `403`). Тело запроса использует `X-User-Id` как единственный источник субъекта; если эндпоинт принимает `userId` в теле — он обязан совпадать с `X-User-Id` (иначе `403` по тому же принципу изоляции) либо не дублировать его. Вызовы `require_owner` помечаются no-op / удаляются (backend).

### 4. JWT / Apple — «дремлют», НЕ удаляются

- Код встроенного issuer (`src/app/auth/issuer.py`, `JwtVerifier`) и Sign in with Apple (`src/app/auth/apple.py`, [ADR-043](ADR-043-sign-in-with-apple.md)) **не удаляется**. Роутеры `/v1/auth/*` могут оставаться зарегистрированными, но **клиент их не использует** для авторизации `/v1/*`.
- БД-схема не трогается: таблицы `auth_devices`/`auth_refresh_tokens`/`auth_identities` (миграции `0005`/`0012`) сохраняются, их тесты остаются зелёными. [ADR-018](ADR-018-embedded-auth-issuer.md)/[ADR-043](ADR-043-sign-in-with-apple.md) → статус **«дремлет»** (not retired): решения остаются валидными как путь будущего апгрейда идентичности, но не на горячем клиентском пути.
- Возврат к JWT-модели или гибрид (JWT поверх клиентского ключа) — аддитивен, отдельным ADR.

### 5. Swagger security-схемы

- `bearerAuth` (HTTP Bearer JWT) для клиентского контура заменяется **двумя `APIKeyHeader`-схемами**:
  - `clientApiKey` — `type: apiKey`, `in: header`, `name: X-API-Key` — клиентская авторизация всех `/v1/*`.
  - `userId` — `type: apiKey`, `in: header`, `name: X-User-Id` — идентичность субъекта (объявляется, чтобы тестировщик подставлял оба заголовка через Authorize).
- `adminToken` (`X-Admin-Token`, `APIKeyHeader`) — **без изменений** ([ADR-009](ADR-009-admin-token-auth.md)).
- `adaptyWebhook` (HTTP Bearer) — без изменений ([ADR-029](ADR-029-adapty-subscription-webhook.md)).
- `bearerAuth` (`HTTPBearer`) для клиентского пути снимается с операций `/v1/*`; объявление scheme может остаться в коде на время «спящего» JWT (не навешивается на клиентские операции). Привязка к каждому эндпоинту — ровно одна корректная (`clientApiKey`+`userId` / `adminToken` / `adaptyWebhook` / none). Стандарт — [08-api-documentation.md §R2](../08-api-documentation.md).
- **Две клиентские схемы — `AND` (оба заголовка обязательны ВМЕСТЕ), не `OR`.** Контракт: каждый клиентский `/v1/*` требует **одновременно** валидный `X-API-Key` **и** `X-User-Id`; ни один из них в отдельности доступа не даёт (`401` при отсутствии любого). В OpenAPI это выражается **одним** security-requirement-объектом с обоими ключами: `security: [{ "clientApiKey": [], "userId": [] }]` (ключи внутри одного объекта = логическое AND).
- **Требуется post-process `app.openapi()` (custom_openapi в `src/app/main.py`).** Дефолт FastAPI при двух `SecurityBase`-зависимостях на операции публикует их как **раздельные** объекты `security: [{ "clientApiKey": [] }, { "userId": [] }]` — это логическое **OR** (достаточно одной схемы), что **противоречит** контракту. Поэтому реализация обязана пост-обработать сгенерированную схему: для каждой клиентской операции `/v1/*` **слить** пару раздельных объектов в один AND-объект `[{ "clientApiKey": [], "userId": [] }]`. Объекты `adminToken` / `adaptyWebhook` / public — не трогаются. Без этого слияния Swagger/контракт декларируют OR — это дефект (reviewer/qa обязаны отвергнуть OR-вариант). Канон представления — [08-api-documentation.md §R2/§R4](../08-api-documentation.md).

## Consequences

**Положительные:**
- Простейшая клиентская модель: один заголовок-ключ + один заголовок-идентичность; не нужен issuer/JWKS/refresh-rotation на горячем пути.
- Изоляция от admin сохраняется by construction (разные секреты, заголовки, зависимости — [ADR-009](ADR-009-admin-token-auth.md)).
- БД-схема и тесты auth не ломаются (JWT/Apple дремлют, не удалены).
- Lazy provisioning ([ADR-007](ADR-007-lazy-user-provisioning.md)) переиспользуется без изменений логики.

**Отрицательные / ограничения:**
- `CLIENT_API_KEY` — высокопривилегированный единый секрет (компрометация = доступ ко всем `userId`). Нет криптографической привязки `X-User-Id` к субъекту (доверие к клиенту). Усиление — [Q-044-1](../99-open-questions.md).
- Идентичность больше не самоверифицируема сервером (нет подписи токена) — ответственность за корректность `X-User-Id` смещена на доверенный клиент.
- В коде остаётся неиспользуемый на горячем пути JWT/Apple-код (осознанный, ради нерушимости схемы/тестов; не tech-debt — путь апгрейда).

## Alternatives

1. **Оставить JWT (ADR-018) как клиентский контур.** Отвергнуто решением пользователя: для control-plane-клиента, который сам владеет идентичностью пользователей, issuer/refresh-rotation избыточны; единый доверенный ключ проще.
2. **Удалить JWT/Apple-код.** Отвергнуто: сломало бы БД-схему (`auth_*` таблицы) и зелёные тесты, закрыло бы путь апгрейда идентичности. «Дремлет» строго лучше.
3. **Подписывать `X-User-Id` HMAC под клиентским ключом (доказательство владения identity).** Сильнее (нельзя подставить произвольный `userId` без ключа — но ключ и так даёт это право). Отложено как усиление [Q-044-1](../99-open-questions.md): на старте доверие к клиенту достаточно.
4. **Per-client ключи (несколько `CLIENT_API_KEY`) с атрибуцией.** Избыточно при единственном доверенном клиенте (iOS control plane); путь — [Q-044-1](../99-open-questions.md).
