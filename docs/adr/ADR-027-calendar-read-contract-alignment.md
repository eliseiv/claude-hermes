# ADR-027 — Унификация контракта `calendar.read` с `calendar.create_events` (имена `start`/`end` + ISO8601 datetime)

- **Статус:** Accepted
- **Дата:** 2026-06-11
- **Связано:** [ADR-011](ADR-011-server-side-tools.md) (классы tools: client-side), [ADR-019](ADR-019-tools-catalog-endpoint.md) (каталог `/v1/tools`), [ADR-024](ADR-024-history-payload-domain-normalization.md) (нормализация payload истории), [ADR-026](ADR-026-global-server-side-tools-and-time-now.md) (`time.now` — источник текущей даты для модели)
- **Затрагивает:** `src/app/chat/tools.py` (`CalendarReadArgs`, `TOOL_DESCRIPTIONS`), `docs/modules/chat-orchestrator/02-api-contracts.md`, `docs/API-REFERENCE.md`
- **Характер изменения:** **breaking change публичного iOS-контракта** инструмента `calendar.read` (требует обновления iOS-клиента).

## Context

Два связанных **client-side** ([ADR-011](ADR-011-server-side-tools.md)) инструмента календаря имели **расходящийся контракт** аргументов диапазона:

| Инструмент | Имена аргументов диапазона | Формат значения (де-факто, что генерировала модель) |
|---|---|---|
| `calendar.read` | `startDate` / `endDate` | date-only (`"2026-06-11"`) |
| `calendar.create_events` | `start` / `end` (внутри `events[]`) | ISO8601 datetime (`"2024-01-13T00:00:00"`, naive local) |

**Симптом (репорт iOS-тестировщика).** При запросе событий «за один день» модель подставляла в `calendar.read` одну и ту же date-only-строку в `startDate` и `endDate` (`startDate == endDate`), и **невозможно задать диапазон по времени внутри дня** (например «с 09:00 до 18:00 сегодня»). Инструмент чтения не мог выразить интервал тоньше суток.

**Корневая причина (подтверждена чтением кода).** Контракт аргументов client-side инструментов коммуницируется модели **только** через:
1. Pydantic-схему args (`CalendarReadArgs`) → JSON Schema (`tool_input_schema()`), где `startDate`/`endDate` объявлены как простой `str` **без формата и без `description`**;
2. человекочитаемое `TOOL_DESCRIPTIONS[calendar.read]` = `"Read calendar events in a date range."` — формат значения **не указан**.

При этом нормативная документация (`docs/modules/chat-orchestrator/02-api-contracts.md`) уже декларировала args `calendar.read` как `{ "startDate": "ISO8601", "endDate": "ISO8601" }` — то есть **docs опережали код** (формат «ISO8601» был в docs, но не доводился до модели ни схемой, ни описанием). Без явного указания формата модель выбирала кратчайшее валидное представление даты — date-only.

Дополнительно — расхождение **имён** (`startDate`/`endDate` vs `start`/`end`) повышало вероятность ошибок модели и усложняло клиентскую реализацию (две разные конвенции для одной и той же семантики «интервал времени»).

> Примечание: `calendar.read`/`calendar.create_events` — это **простые `str`-аргументы**, формат значения по контракту client-side tools коммуницируется модели через описание/схему, а **не** валидируется backend как datetime (см. [ADR-011](ADR-011-server-side-tools.md): backend только инициирует tool-call, исполняет iOS). Решение о вводе/невводе серверной валидации формата — отдельный пункт ниже (Decision §3).

## Decision

Привести `calendar.read` к **полной консистентности** с `calendar.create_events`: идентичные имена аргументов диапазона и идентичный формат времени.

### 1. Переименование аргументов (breaking)
`calendar.read` args: `startDate` → **`start`**, `endDate` → **`end`**. Поле `calendarId: str | None = None` сохраняется без изменений.

Итоговая схема args `calendar.read`:
```json
{ "start": "ISO8601 datetime", "end": "ISO8601 datetime", "calendarId": "string (optional)" }
```
Идентична конвенции имён `calendar.create_events` (`events[].start` / `events[].end`).

### 2. Формат значения — ISO8601 datetime, идентичный `create_events`
`start`/`end` в `calendar.read` принимают **ISO8601 datetime** в том же виде, что `calendar.create_events`: **naive local datetime** без offset, секундная точность — например `"2026-06-11T09:00:00"`. **Без date-only backward-compat** (строки вида `"2026-06-11"` больше не являются целевым контрактом).

Обоснование выбора *naive local* (а не tz-aware): это **уже сложившийся де-факто формат** `calendar.create_events` (`"2024-01-13T00:00:00"`), который iOS успешно использует; календарь iOS оперирует локальным временем устройства; tz-aware-переход потребовал бы менять **оба** инструмента и контракт create без необходимости. Цель ADR — *выровнять read под create*, а не реформировать формат create. Tz-aware как возможное будущее усиление обоих инструментов — [Q-027-1](../99-open-questions.md) (Open, не блокирует).

### 3. Валидация формата — НЕ вводится (симметрично read и create)
Аргументы `start`/`end` остаются простым `str` (как сейчас у `create_events` и как было у `read`). **Серверная datetime-валидация формата НЕ вводится** — ни для `read`, ни для `create`. Обоснование:
- симметрия: единый подход к обоим инструментам (требование пользователя);
- client-side контракт ([ADR-011](ADR-011-server-side-tools.md)): фактическое исполнение и парсинг datetime — на стороне iOS (EventKit); backend лишь инициирует tool-call;
- минимально-достаточное изменение: ввод строгой Pydantic-валидации datetime сделал бы невалидную строку причиной `422` всего хода (обрыв генерации), тогда как сейчас модель/клиент обрабатывают формат мягко;
- формат доводится до модели через **обновлённые описания** (Decision §4) — этого достаточно, чтобы модель генерировала datetime.

Если в будущем потребуется строгая валидация — её вводят **симметрично** для `read` и `create` отдельным ADR, с явным выбором поведения при ошибке (tool-result error vs `422`).

### 4. Описания для модели (`TOOL_DESCRIPTIONS`) — формат закрепляется в тексте
Чтобы модель генерировала `start`/`end` в ISO8601 datetime, описания обоих инструментов делаются **самодостаточными по формату** (формат не должен жить только в docs — он должен доходить до модели):

- `calendar.read` →
  `"Read calendar events within a time range. 'start' and 'end' are ISO8601 datetime strings in local time without timezone offset, e.g. '2026-06-11T09:00:00'. For a whole day use start at 00:00:00 and end at the next day 00:00:00 (end-exclusive). Use the time.now tool if you do not know the current date."`
- `calendar.create_events` → описание дополняется тем же явным указанием формата `start`/`end` (ISO8601 datetime, local, no offset, пример `'2026-06-11T09:00:00'`), чтобы read и create несли **идентичную** формулировку формата.

(Точные backend-указания — в `02-api-contracts.md`; backend-исполнитель формулирует финальные строки в рамках этого ADR.)

### 5. Семантика диапазона «события за один день» — end-exclusive (полночь следующего дня)
Единая конвенция для модели и iOS: **`start` = начало интервала включительно, `end` = граница интервала исключительно (end-exclusive)**. «Весь день D» = `start = "D T00:00:00"`, `end = "D+1 T00:00:00"` (полночь следующего дня).

Обоснование выбора end-exclusive (а не `end = 23:59:59`):
- однозначно покрывает события, начинающиеся ровно в полночь следующего дня = не включаются (нет двойного учёта при смежных днях);
- нет «дыры» в одну секунду между `23:59:59` и `00:00:00`;
- полуинтервал `[start, end)` — стандартная и наименее ошибочная конвенция для временных диапазонов;
- единообразно масштабируется на многодневные диапазоны.

Конвенция фиксируется в описании инструмента (Decision §4), чтобы модель её применяла; фактическая фильтрация событий по интервалу — ответственность iOS (EventKit) в пределах этого полуинтервала.

### 6. Каталог инструментов и количество tools — без изменений количества
Каталог остаётся **14 инструментов** ([ADR-019](ADR-019-tools-catalog-endpoint.md)/[ADR-026](ADR-026-global-server-side-tools-and-time-now.md)). Меняется **только `inputSchema`** записи `calendar.read` в `GET /v1/tools` (поля `start`/`end` вместо `startDate`/`endDate`) — автоматически, т.к. каталог генерируется из `_ARGS_BY_TOOL` (single source of truth). `calendar.read` остаётся `execution=client`, `mutating=false`. BUG-3 name-map (`_DOMAIN_TO_ANTHROPIC`) **не затрагивается** — это маппинг **имён инструментов** (`calendar.read ↔ calendar_read`), не имён аргументов.

## Consequences

**Положительные:**
- единый контракт диапазона для обоих календарных инструментов (`start`/`end`, ISO8601 datetime, naive local) — меньше ошибок модели, проще iOS-реализация;
- решён исходный баг: достижим **диапазон по времени внутри дня** (например 09:00–18:00) и однозначная конвенция «весь день» (end-exclusive);
- формат теперь **доходит до модели** через описание (устранён корневой разрыв docs↔модель);
- результат `calendar.read` (формат `events[].start`/`events[].end` = ISO8601) и create согласованы.

**Отрицательные / риски:**
- **Breaking change публичного iOS-контракта** `calendar.read`: iOS-клиент, исполняющий `calendar.read`, обязан читать аргументы `start`/`end` (а не `startDate`/`endDate`) и интерпретировать значения как datetime. Без обновления клиента старая реализация перестанет находить ключи args → ошибка исполнения tool на устройстве. **Требуется скоординированный релиз iOS.**
- Исторические `chat_steps`/`tool_calls` хранят то, что было сгенерировано на момент записи (старые `startDate`/`endDate`-вызовы). При реплее/просмотре истории (`GET /v1/chats/{id}`) старые шаги отдаются **как есть** — нормализация ([ADR-024](ADR-024-history-payload-domain-normalization.md)) не переписывает payload и не трогает имена аргументов внутри `tool_use.input`. Влияние ограничено отображением истории; **миграция сохранённых сессий не требуется** (см. [Q-027-2](../99-open-questions.md), решено: миграции нет).

## Alternatives

1. **Оставить `startDate`/`endDate`, только закрепить формат datetime в описании** (не переименовывать). Отклонено: сохраняет рассинхрон имён с create — пользователь явно потребовал полную консистентность имён; источник ошибок модели остаётся.
2. **Date-only с дополнительным полем времени** (`startTime`/`endTime`). Отклонено: усложняет схему, расходится с create (у create — единое datetime-поле), не даёт консистентности.
3. **Сохранить backward-compat: принимать и date-only, и datetime** в `read`. Отклонено: пользователь явно выбрал «строго datetime, без date-only backward-compat»; двусмысленность формата — источник того же бага.
4. **Перевести оба инструмента на tz-aware ISO8601** (с offset). Отклонено в этом ADR: ломает де-факто формат create без необходимости; цель — выровнять read под существующий create, не реформировать create. Оставлено как [Q-027-1](../99-open-questions.md).
5. **`end = 23:59:59` для «весь день»** (inclusive). Отклонено в пользу end-exclusive (полночь следующего дня) — см. Decision §5.
6. **Ввести строгую serverside datetime-валидацию args.** Отклонено: см. Decision §3 (асимметрия с create, обрыв хода на `422`, противоречит client-side-модели ADR-011).
