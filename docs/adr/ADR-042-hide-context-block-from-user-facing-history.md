# ADR-042 — Скрытие conversation-settings блока (ADR-037) из user-facing вывода истории и превью

- **Статус:** Accepted
- **Дата:** 2026-06-19
- **Тип:** implementation-ADR (read-time strip; без миграции, без изменения хранения/генерации)
- **Ревизует:** [ADR-037](ADR-037-chatrunrequest-context-allowlist-injection.md) **только в части отображения** блока в истории/превью (тело ADR-037 неизменно — см. пометку-ревизию в нём). Хранение и инъекция в промт (ADR-037 §4–§5) **не меняются**.
- **Связанные:** [ADR-024](ADR-024-history-payload-domain-normalization.md) (нормализация payload истории на границе сериализации — тот же паттерн «read-time, на копии, хранение не мутируется»), [ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md) (персист user-step), [ADR-039](ADR-039-optional-message-with-attachments.md) (image-only/file-only ход, edge с context), [ADR-033](ADR-033-llm-provider-abstraction.md) (провайдер-агностичность)

## Context

По [ADR-037](ADR-037-chatrunrequest-context-allowlist-injection.md) при наличии `ChatRunRequest.context` backend строит детерминированный текст-блок `_render_context_block(context)` →

```
[Conversation settings for this message: codeLanguage=Swift; responseStyle=concise; locale=ru-RU]
```

и **лидирующе** склеивает его с текстом пользователя через `_compose_turn0_text(block, msg)` → `f"{block}\n\n{msg}"` (`src/app/chat/orchestrator.py`). Изменённый текст **персистится** как первый text-блок user-шага в `chat_steps.payload.content` (корректный replay — ADR-037 §4).

**Репорт iOS-разработчика:** поскольку блок персистится внутри текста user-сообщения, он **виден пользователю** в двух местах:

1. История чата `GET /v1/chats/{id}` — `steps[].payload.content[0].text` начинается со служебного блока;
2. Превью чата в списке `GET /v1/chats` — превью формируется в `ChatsRepository._preview` → `_text_from_payload` (`src/app/chats/repository.py`), который берёт текст из `content` последнего user/assistant-шага; для user-шага с `context` превью начинается с `[Conversation settings for this message: …]`. (`_text_summary` в `src/app/chats/service.py` используется **только** для assistant-шагов в steps-view `/v1/chats/{id}/steps` и user-текст превью **не** формирует.)

Это служебная обвязка промта (на английском, формат сервера), не текст пользователя. Показывать её в UI некорректно.

**Согласованное с пользователем решение — вариант B (read-time strip):** срезать ведущий settings-блок **только при отдаче** (история + превью). Хранимое `chat_steps.payload` **не менять** — модель на реплее (`_build_messages`) по-прежнему получает блок, поведение генерации/контекста для модели не меняется. Без миграции.

### Отклонённые варианты

- **Вариант A — не персистить блок в payload (хранить только текст пользователя), реинъектировать блок на реплее.** Отклонён: менял бы хранение и логику `_build_messages`, требовал бы реинъекции блока из отдельно хранимого `context` (которого нет — ADR-037 намеренно не хранит `context`), риск рассинхронизации хранимого ↔ реплея, потенциальная миграция для уже накопленных историй. Read-time strip проще и безопаснее.
- **Вариант C — миграция: переписать накопленные payload, вырезав блок.** Отклонён: разрушает replay (модель перестанет получать блок на уже существующих ходах → меняет поведение генерации старых сессий), необратимо, дорого. Противоречит инварианту ADR-037 §4 (блок — часть хода для replay).
- **Скрывать на стороне iOS (клиент сам срезает префикс).** Отклонён: формат блока — деталь backend; завязывать клиент на точный серверный формат служебной строки хрупко. Один источник истины формата — backend, он же его и срезает.

## Decision

### 1. Read-time strip — только при отдаче, хранение неизменно

Срез ведущего settings-блока применяется **исключительно на границе сериализации** ответов:

- история `GET /v1/chats/{id}` — в user-facing преобразовании payload (`ChatsService._normalize_payload`, `src/app/chats/service.py`, на **глубокой копии**, как уже делает ADR-024);
- превью `GET /v1/chats` — в `ChatsRepository._preview` / `_text_from_payload` (`src/app/chats/repository.py`), к тексту первого `text`-блока user-шага, **строго ДО** `_truncate`/collapse (см. §2-порядок). **НЕ** в `_text_summary` — он формирует только assistant-шаги steps-view, не превью.

`chat_steps.payload` (хранимое) и `_build_messages` (реплей модели, `orchestrator.py`) **НЕ меняются**. Инвариант ADR-037 §4 сохраняется: на continuation/resume/replay модель получает текст с блоком ровно как прежде. Поведение генерации и контекст для модели не меняются. **Миграции нет.**

### 2. Общий helper — единственный источник истины среза

Ввести **один** helper (например `strip_context_block(text: str) -> str`), переиспользуемый обоими call-site: **история** (`ChatsService._normalize_payload`, `src/app/chats/service.py`) и **превью** (`ChatsRepository._preview` / `_text_from_payload`, `src/app/chats/repository.py`). Дублировать regex/логику в двух местах запрещено (рассинхрон формата). Helper должен быть один, импортируемый обоими модулями (рекомендуется общий util или экспорт из одного модуля; backend выбирает точное место).

Helper срезает блок **строго в начале** текста; если блока нет — возвращает текст без изменений (no-op).

**Порядок применения в превью (критично).** Якорь среза (§3) требует завершающего `\n\n`. `_truncate` в `repository.py` (`" ".join(value.split())`) **схлопывает** `\n\n` в один пробел, поэтому после collapse якорь не сматчится и блок останется. Следовательно `strip_context_block` ОБЯЗАН применяться к **сырому** тексту первого text-блока user-шага **строго ДО** `_truncate`/collapse. Порядок: `strip_context_block(raw_text)` → затем `_truncate(stripped)`. В истории такой проблемы нет — `_normalize_payload` не усекает/не схлопывает текст, strip применяется к полному тексту блока.

### 3. Точный матч (сентинел) — якорь только в начале, только серверный формат

Срезать строго по якорю в начале строки, точно соответствующему формату, который сервер сам генерирует в `_render_context_block`:

```
^\[Conversation settings for this message: [^\]]*\]\n\n
```

- **`^`** — только в самом начале текста (блок ADR-037 всегда лидирует, ADR-037 §4). Совпадение в середине текста пользователя **не** трогается.
- **`\[Conversation settings for this message: `** — точный литеральный префикс служебной строки (детерминированный, ADR-037 §3).
- **`[^\]]*`** — содержимое блока до закрывающей `]`. Внутри значений `]` не встречается: для свободных строк (`codeLanguage`/`tone`) backend экранирует/санитизирует значение (ADR-037 §3 — `;`,`=`,переводы строк → пробел), `locale` ограничен классом `[A-Za-z0-9_-]`, enum-ключи закрыты. Поэтому `[^\]]*` корректно захватывает ровно одно содержимое блока до его закрывающей скобки.
- **`\]\n\n`** — закрывающая скобка и ровно тот разделитель `\n\n`, что вставляет `_compose_turn0_text`. После среза остаётся чистый `message`.

Это **минимизирует риск задеть реальный текст пользователя**: совпадает только точный сервер-генерируемый префикс в начале. Сообщение пользователя, начинающееся с похожего, но не совпадающего текста, или содержащее `[Conversation settings…]` не в начале, не затрагивается.

### 4. Скоуп — только user-шаги; только при наличии блока

- Срез применяется **только к шагам `role="user"`** (блок ADR-037 инъектируется исключительно в turn-0 user-сообщение). Шаги `role="assistant"` и `role="tool"` **не трогаются** — там блока нет, и любое совпадение было бы ложным.
- Срез применяется к **первому (лидирующему) text-блоку** user-шага (там и только там стоит блок). Прочие блоки user-шага (attachment-плейсхолдеры) не трогаются.
- **Нет блока → no-op.** Если текст не начинается с якоря (user не присылал `context`, либо после валидации не осталось валидных ключей — `_render_context_block` вернул `None` и блок не вставлялся), helper возвращает текст без изменений.

### 5. Edge: image-only / file-only ход с context

По [ADR-039 §3](ADR-039-optional-message-with-attachments.md) + ADR-037: если `message` пуст, а `context` валиден, `_compose_turn0_text` возвращает **только блок** (`block or ""`, без висячего `\n\n`). То есть единственный text пользователя = служебный блок.

Однако якорь среза (§3) требует завершающего `\n\n`, которого в этом случае нет. Чтобы корректно обработать и этот вариант, helper **дополнительно** срезает блок-без-хвоста, когда он составляет **весь** текст:

- паттерн полного-совпадения: `^\[Conversation settings for this message: [^\]]*\]$` (весь текст = блок) → результат пустая строка `""`.

После среза текст user-шага **пустой**. В истории это корректно: исходное сообщение пользователя было **без текста** (image-only / file-only ход), показывается **пустой текст + attachment-плейсхолдеры**. Превью такого чата (`_text_summary`) — пустая строка для этого шага (как и для любого исходно-текстопустого сообщения). Это ожидаемое поведение, не баг.

Порядок применения в helper: сначала пытаться срезать «блок + `\n\n` + текст» (§3), при отсутствии хвоста — «блок == весь текст» (§5). Оба — только в начале/целиком, регистронезависимости нет (формат фиксирован).

### 6. Инварианты

- **Хранение неизменно:** `chat_steps.payload` не мутируется (срез — на копии при отдаче, как ADR-024). Без миграции.
- **Реплей модели неизменен:** `_build_messages` отдаёт модели текст с блоком как прежде → генерация/контекст для модели не меняются. Поведение ADR-037 для модели полностью сохранено.
- **Провайдер-агностично:** срез — над доменным текстом при отдаче, не зависит от провайдера (Anthropic/OpenAI, [ADR-033](ADR-033-llm-provider-abstraction.md)).
- **Биллинг неизменен** (1 кредит, [ADR-006](ADR-006-credit-billing-and-subscription-grant.md)).
- **Безопасность:** срез не раскрывает ничего нового; наоборот, перестаёт показывать служебную обвязку. Provider id / нормализация ADR-024 не затрагиваются (срез работает по `text`-блокам, ADR-024 — по `tool_use`/`tool_result`; они независимы и совместимы).

## Consequences

**Плюсы.** Пользователь видит в истории и превью только свой текст; служебная обвязка промта скрыта. Без миграции, без изменения хранения и логики генерации (нулевой риск для replay/контекста модели). Один helper — единственный источник истины формата среза, симметрично ADR-024 (read-time, на копии). Провайдер-агностично.

**Минусы / границы.** Helper завязан на точный формат `_render_context_block` (ADR-037 §3) — при изменении формата блока надо синхронно обновить и якорь среза (оба в одном коде, риск низкий, тест-инвариант: «то, что вставил `_render_context_block`, должен срезать `strip_context_block`»). Накопленные истории старых ходов на реплее по-прежнему несут блок (так и задумано — replay не меняется), но в user-facing выводе он теперь скрыт независимо от давности хода (strip применяется ко всем отдаваемым шагам).

## Указания backend (implementation)

1. **Helper (один, единственный источник истины среза).** Добавить функцию `strip_context_block(text: str) -> str` (имя ориентировочное) в общий util/модуль, **импортируемый обоими** call-site — `src/app/chats/service.py` (история) и `src/app/chats/repository.py` (превью). Helper должен быть один (не дублировать в двух файлах). Логика:
   - если `text` матчит `^\[Conversation settings for this message: [^\]]*\]\n\n` → вернуть `text` без этого префикса (всё, что после `\n\n`);
   - иначе если `text` матчит **целиком** `^\[Conversation settings for this message: [^\]]*\]$` (edge §5, image-only с context) → вернуть `""`;
   - иначе → вернуть `text` без изменений (no-op).
   - Скомпилировать regex один раз на модуль (module-level `re.compile`). Не логировать содержимое (значения `context` не логируются, ADR-037 §6 / [05-security.md](../05-security.md)).
2. **История — `ChatsService._normalize_payload`** (`src/app/chats/service.py`). Применять helper **только** когда `step.role == "user"`, к **первому** `text`-блоку `payload["content"]` (тому, где `block["type"] == "text"`), на той же **глубокой копии**, что уже делает ADR-024 (хранение не мутировать). assistant/tool-шаги — не трогать. `tool_use`/`tool_result`-нормализация ADR-024 — без изменений (они независимы).
3. **Превью — `ChatsRepository._preview` / `_text_from_payload`** (`src/app/chats/repository.py`). Превью списка `GET /v1/chats` формируется именно здесь (`ChatsService.list_chats` берёт готовый `item.preview` из репозитория), **НЕ** в `_text_summary` (тот формирует только assistant-шаги steps-view). `_preview` выбирает последний user/assistant-шаг → `_text_from_payload(step.payload)` → `_truncate`. Срез нужен **только** когда выбранный шаг — `role="user"`, к тексту **первого** `text`-блока. **Критично — порядок:** применить `strip_context_block` к **сырому** тексту первого user-text-блока **ДО** `_truncate` (collapse схлопывает `\n\n` → пробел и ломает якорь, см. Decision §2). Чистый вариант: в `_preview` различать роль выбранного шага (`step.role`) и для user применять `strip_context_block` к лидирующему text-блоку до `_truncate`; assistant-шаг не трогать. **Инвариант — превью user-сообщения не должно начинаться со служебного блока**, не-user превью не трогаются. **НЕ** добавлять срез в `_text_summary` (Decision §1 пометка).
4. **НЕ трогать** `_render_context_block`, `_compose_turn0_text`, `_build_messages`, персист в `chat_steps.payload` (orchestrator.py). Генерация/контекст для модели — без изменений.
5. **Скоуп строго:** только `role="user"`, только лидирующий text-блок, только при совпадении якоря. Никаких изменений для assistant/tool/прочих блоков.
6. Тест-инвариант для qa: для любого валидного `context`, дающего непустой блок, `strip_context_block(_compose_turn0_text(_render_context_block(context), msg)) == msg` (включая `msg == ""` → `""`); и для текста без блока — strip = identity.
