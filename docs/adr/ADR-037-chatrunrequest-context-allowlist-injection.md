# ADR-037 — `ChatRunRequest.context`: per-message allowlist + инъекция в user-сообщение

- **Статус:** Accepted (ревизия отображения 2026-06-19 → [ADR-042](ADR-042-hide-context-block-from-user-facing-history.md))
- **Дата:** 2026-06-17
- **Связанные:** [ADR-012](ADR-012-assistant-mode-vs-billing-mode.md) (assistant_mode), [ADR-020](ADR-020-inline-base64-attachments-mvp.md) (turn0 user-content), [ADR-021](ADR-021-deterministic-step-order-and-block-normalization.md) (персист user-step), [ADR-033](ADR-033-llm-provider-abstraction.md) (провайдер-агностичность), [ADR-036](ADR-036-workspaces-implementation.md) (инъекция workspace.instructions — образец, но в system)

> **Ревизия отображения (2026-06-19, [ADR-042](ADR-042-hide-context-block-from-user-facing-history.md)).** §4 фиксирует, что settings-блок персистится внутри текста user-шага в `chat_steps.payload` (для корректного replay) — это сохраняется без изменений. Но из-за этого блок был **виден в user-facing выводе**: в истории `GET /v1/chats/{id}` (`steps[].payload.content[].text`) и в превью `GET /v1/chats`. [ADR-042](ADR-042-hide-context-block-from-user-facing-history.md) **ревизует ADR-037 только в части отображения**: блок срезается **при отдаче** истории и превью (read-time strip, единый helper). Хранение (§4), инъекция в промт и реплей модели (`_build_messages`) этого ADR **не меняются** — модель по-прежнему получает блок. Тело ADR-037 ниже остаётся в силе как есть.

## Context

Поле `ChatRunRequest.context: dict[str, Any] | None` (`src/app/schemas/chat.py`) сегодня **только** валидируется по размеру (`size_limit_context`, ≤64KB сериализованного JSON → `422`) и **игнорируется**: оно не прокидывается в `orchestrator.run()` (в сигнатуре `run()` параметра нет), не влияет на промт, не персистится отдельно.

iOS-разработчику нужно передавать на **каждый** запрос дополнительные настройки разговора, которые могут меняться по ходу чата: язык программирования для code-режима, желаемый стиль/тон ответа, локаль и т.п. Это **не** атрибут сессии (как `mode`/`assistantMode`/`model`/`projectId`/`workspaceProjectId`, которые фиксируются при создании), а характеристика **конкретного хода**.

Принятые с пользователем решения (обязательны к реализации):

1. **Per-message, не session-fixed.** `context` присылается и применяется на каждом `/chat/run`, может меняться от хода к ходу. Поэтому он **не хранится** в `chat_sessions` и **миграция БД не нужна** — context влияет на содержимое текущего user-сообщения, которое и так персистится как user-step в `chat_steps` (корректный replay).
2. **Allowlist ключей** — фиксированный набор известных ключей с per-key валидацией типа/длины; неизвестные ключи игнорируются (forward-compat).
3. **Инъекция в user-сообщение, не в system-prompt** — чтобы не ломать prompt-кэш (system+tools стабильны; `cache_control: ephemeral` на system в `anthropic_client._build_system`) и по безопасности: context = собственные данные пользователя, не должен получать авторитет системного промта.

## Decision

### 1. Allowlist ключей и per-key валидация

Известный набор ключей `context` (всё остальное игнорируется). Значения нормализуются `strip()`; пустое после strip → ключ игнорируется. Значение неверного типа или превышающее лимит длины → **этот ключ игнорируется** (lenient, см. §2). Длина — в символах после `strip` (Unicode codepoints).

| Ключ | Тип | Валидация | Семантика |
|---|---|---|---|
| `codeLanguage` | str | `strip`, непустой, ≤40 символов | Предпочитаемый язык программирования для code-режима (`assistantMode=code`). Свободная строка (множество языков/диалектов/версий слишком велико для enum). Пример: `Swift`, `Python 3.12`, `TypeScript`. |
| `responseStyle` | str (enum) | `strip`, нормализация к lower-case, **должно входить в `{concise, balanced, detailed}`**; иначе ключ игнорируется | Общая форма ответа. Закрытый enum: набор стабилен, легко рендерится и предсказуем для модели. |
| `verbosity` | str (enum) | `strip`, lower-case, **должно входить в `{low, medium, high}`**; иначе ключ игнорируется | Степень детализации/длины. Закрытый enum (отдельно от `responseStyle`: style = форма, verbosity = объём). |
| `tone` | str | `strip`, непустой, ≤40 символов | Тон ответа (свободная строка: `friendly`, `formal`, `neutral`, `professional` и пр. — слишком вариативно для enum). |
| `locale` | str | `strip`, непустой, ≤35 символов, набор символов `[A-Za-z0-9_-]` (BCP-47-подобный); иначе ключ игнорируется | Предпочитаемый язык/локаль ответа. Пример: `ru-RU`, `en`, `de-DE`. Символьный класс ограничен, чтобы исключить инъекцию произвольного текста через «локаль». |

**Обоснование enum vs свободная строка.** `responseStyle` и `verbosity` — закрытые enum: малый стабильный набор, который backend контролирует, рендерит предсказуемо и не пропускает произвольный текст пользователя в промт. `codeLanguage`/`tone`/`locale` — свободные строки (с лимитом длины и, для `locale`, символьным классом): пространство значений велико/открыто, перечислить нельзя; лимит длины + (для locale) символьный класс ограничивают поверхность инъекции.

Минимально покрытые dev-кейсы из требования: **язык программирования** (`codeLanguage`) и **стиль ответа** (`responseStyle`). Эти два ключа — обязательная часть поставки; остальные три (`verbosity`/`tone`/`locale`) входят в тот же allowlist.

Расширение allowlist (доп. ключи, перевод свободных строк в enum, i18n) — [Q-037-1](../99-open-questions.md). Связь с `preferences.code_defaults` (серверные дефолты vs per-message override) — вне scope, [TD-028](../100-known-tech-debt.md).

### 2. Поведение на невалидное (двухуровневая валидация)

- **Уровень схемы (без изменений):** существующая size-валидация `_check_sizes` сохраняется. Грубо-битое / огромное тело `context` (>`size_limit_context`, дефолт 64KB) → **`422`** до оркестрации. Тип поля остаётся `dict[str, Any] | None` (Pydantic StrictModel; не-объект → `422` как сейчас).
- **Уровень рендера (новое, lenient):** при сборке блока неизвестные ключи **игнорируются**; ключ с неверным типом/длиной/вне-enum/вне-символьного-класса значением — **этот ключ игнорируется**, остальные применяются. Запрос **не** падает из-за «частично невалидного» context. Это forward-compat (iOS может слать новые ключи старому backend) и устойчивость (один битый ключ не ломает ход).

### 3. Рендер-формат — детерминированный компактный блок

Helper `_render_context_block(context: dict[str, Any] | None) -> str | None` в `orchestrator.py`:

- Возвращает `None`, если `context` отсутствует, пуст, или после валидации не осталось ни одного валидного ключа (→ поведение полностью как без `context`).
- Иначе собирает блок с **фиксированным порядком ключей** (порядок allowlist-таблицы: `codeLanguage, responseStyle, verbosity, tone, locale`), включая только валидные присутствующие ключи:

```
[Conversation settings for this message: codeLanguage=Swift; responseStyle=concise; locale=ru-RU]
```

- Значения после `strip`; для свободных строк (`codeLanguage`/`tone`/`locale`) защитная обрезка по лимиту длины ключа (страховка). Разделители `; ` и `=` экранируются заменой в значениях (для свободных строк `;` и `=` и переводы строк → пробел), чтобы значение пользователя не нарушало структуру блока.
- Блок — **обычный текст**, без markdown/JSON, на английском (язык служебной обвязки промта; запрошенный пользователем язык передаётся как значение `locale`).

### 4. Точка и порядок инъекции

- Блок добавляется к содержимому **текущего user-сообщения turn0** в `orchestrator.run()`, в точке сборки `user_payload_content` (где сейчас `[{"type": "text", "text": message}]`).
- **Порядок:** блок-подсказка **лидирует**, затем текст пользователя. Текст первого text-блока становится:

```
<context block>\n\n<message>
```

  (если валидного блока нет — только `message`, как сейчас). Разделитель `\n\n` — по образцу `_system_prompt_with_workspace` (ADR-036).
- Это единственный text-блок user-content (attachments-плейсхолдеры следуют за ним, как сейчас). Изменённый текст **персистится** в `chat_steps.payload` как часть user-step (часть текущего хода) → корректный replay: при continuation/resume блок уже в истории и не подаётся повторно.
- **НЕ** в `system`. **НЕ** на continuation/`/chat/tool-result` (там новый user-message не формируется; context принадлежит только своему ходу и уже в истории).

### 5. Кэш-инвариант и провайдер-агностичность

- `system` и `tools` **не зависят** от `context` → prompt-кэш Anthropic (`cache_control: ephemeral` на system) **не инвалидируется** добавлением context.
- Блок — обычный текст в user-content → работает одинаково на обоих провайдерах ([ADR-033](ADR-033-llm-provider-abstraction.md)): провайдер-специфичная (де)сериализация не затрагивается, orchestrator остаётся провайдер-агностичным.

### 6. Безопасность ([05-security.md](../05-security.md))

- context инъектируется в **user-сообщение**, не в system → не получает авторитет системного промта (нет privilege-escalation через «настройки»).
- Allowlist + per-key лимиты длины + символьный класс для `locale` + enum для `responseStyle`/`verbosity` ограничивают поверхность инъекции произвольного текста.
- Экранирование разделителей не даёт значению сломать структуру служебного блока.
- Содержимое `context` **не логируется** (как и `message`); в логи/метрики попадают максимум имена применённых ключей, не значения.
- Size-guard (`size_limit_context`) сохраняется как защита от раздутого тела.

### 7. Обратная совместимость

- Без поля `context` / пустой объект / нет валидных ключей → `_render_context_block` возвращает `None` → user-сообщение = только `message`. Поведение полностью неизменно.
- Контракт `/v1/chat/run` аддитивен: поле уже существует в схеме, меняется лишь его семантика (с «игнорируется» на «функционально»). iOS, не присылающий `context`, не затронут.
- Биллинг неизменен (1 кредит = 1 сообщение, [ADR-006](ADR-006-credit-billing-and-subscription-grant.md)).

## Consequences

**Плюсы.** Per-message настройки без БД-миграции и без усложнения session-модели; prompt-кэш не ломается; провайдер-агностично; lenient-поведение даёт forward-compat для iOS; безопасность (user-content, не system).

**Минусы / границы.** Модель не обязана строго следовать «настройкам» (это user-hint, не жёсткое ограничение) — приемлемо для UX-настроек. `context` влияет только на ход, в котором прислан (нет «липкости»: чтобы настройка действовала весь чат, iOS шлёт её каждый раздел). Пресет/serverдефолты (`code_defaults`) сюда не сводятся — [TD-028](../100-known-tech-debt.md). Расширение allowlist требует деплоя backend (реестр в коде).

## Alternatives

- **Session-fixed context (как `model`/`assistantMode`).** Отклонено: пользователь требует менять настройки по ходу чата; фиксация лишила бы фичу смысла и потребовала бы колонку/миграцию.
- **Инъекция в system-prompt** (рядом с workspace.instructions, ADR-036). Отклонено: (а) инвалидирует prompt-кэш на каждый ход (system перестаёт быть стабильным); (б) даёт пользовательским данным авторитет системного промта (security).
- **Произвольный pass-through `context` в промт без allowlist.** Отклонено: неконтролируемая инъекция произвольного пользовательского текста с авторитетом «настроек», нет валидации, нет forward-compat-семантики.
- **Жёсткий `422` на любой неизвестный/невалидный ключ.** Отклонено: ломает forward-compat (новый iOS ↔ старый backend) и устойчивость хода; lenient-подход предпочтителен для необязательных UX-настроек.
